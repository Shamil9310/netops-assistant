# NetOps Assistant

NetOps Assistant — внутренний web-сервис для команды сетевых инженеров.

Проект строится как единое рабочее пространство для трёх связанных задач:

- ведение дневного журнала инженера;
- подготовка и сопровождение ночных работ;
- автоматическая генерация отчётности и документов за любой период.

## Что должно уметь приложение

### Дневной контур
- быстрый ввод рабочих активностей;
- фиксация заявок, звонков, работы в TFS, планирования, общения с коллегами, диагностики, документации;
- генерация отчёта за день;
- генерация отчёта за неделю;
- генерация отчёта за произвольный период.

### Контур ночных работ
- создание плана ночных работ;
- разбиение плана по SR / change-блокам;
- хранение шагов, проверок, post-check, rollback и участников;
- использование шаблонов типовых сетевых изменений;
- отметка выполнения шагов в процессе окна;
- формирование итогового отчёта по ночным работам.

### Командный контур
- сотрудники видят и редактируют только свои данные;
- руководитель видит отчёты и данные команды, но не редактирует их;
- разработчик имеет отдельный технический доступ к настройкам, диагностике и системному дашборду.

### Экспорт
Экспорт — обязательная часть продукта.

Обязательные форматы развития:
- Markdown
- TXT
- DOCX
- PDF
- CSV/XLSX

## Технологический стек

### Backend
- FastAPI
- Pydantic Settings
- SQLAlchemy asyncio
- PostgreSQL
- Redis позже, если действительно понадобится очередь/кеш
- LDAP authentication adapter через `ldap3`

### Frontend
- Next.js App Router
- TypeScript
- тёмный enterprise UI

### Инфраструктура
- запуск на одной VM
- Docker Compose на старте
- позже: NGINX reverse proxy, systemd или контейнерный runtime, резервное копирование, observability

### Базовые требования к конфигам
- `NETOPS_ASSISTANT_SECRET_KEY` обязателен для backend во всех окружениях;
- в `production` cookie-сессии должны быть `secure=true`;
- для frontend server-side запросов в контейнере используем `INTERNAL_API_BASE_URL` (например, `http://backend:8000`).

## Целевая VM

Проект планируется к запуску на одной виртуальной машине с лимитом:

- 8 vCPU
- 16 GB RAM
- 128 GB disk

Для MVP и первой production-итерации этого достаточно при умеренном количестве пользователей.

## Контур аутентификации

Продовая цель — LDAP-аутентификация по модели, близкой к тому, как это используется в NetBox:

- проверка пользователя через корпоративный LDAP;
- маппинг LDAP-групп в роли приложения;
- локальный dev-login остаётся только для development режима;
- доступ в систему регулируется ролями и зонами видимости.

## Роли

### employee
Обычный инженер:
- создаёт и редактирует свои записи;
- создаёт свои планы;
- выгружает свои отчёты;
- не видит чужие данные.

### manager
Руководитель:
- видит отчёты и данные команды в режиме чтения;
- не редактирует чужие записи;
- может выгружать отчёт по каждому сотруднику и по группе.

### developer
Разработчик проекта:
- технический и продуктовый доступ;
- доступ к системному дашборду VM;
- доступ к шаблонам, настройкам и сервисным экранам;
- может сопровождать развитие продукта.

## Основные модули

- `journal` — дневной журнал
- `reports` — генерация отчётов
- `plans` — ночные планы работ
- `templates` — шаблоны типовых планов
- `team` — командный доступ и представления руководителя
- `developer` — технический дашборд и сервисные функции
- `export` — выгрузка документов
- `auth` — локальная и LDAP-аутентификация

## Быстрый смысл проекта одной фразой

NetOps Assistant — это рабочая система сетевого инженера, где из единого журнала и планов работ автоматически собираются документы, отчёты и история изменений.
