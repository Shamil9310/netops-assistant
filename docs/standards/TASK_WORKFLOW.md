# TASK_WORKFLOW.md

## Назначение документа

Данный документ определяет обязательный порядок выполнения задач.

Цель — исключить хаотичную разработку и обеспечить:
- предсказуемость изменений;
- контроль архитектуры;
- снижение количества ошибок;
- высокое качество решений.

Любая задача должна выполняться строго по данному процессу.

---

## 1. Анализ

- Понять цель задачи.
- Определить ожидаемый результат.
- Проверить ограничения и контекст.
- Выявить затрагиваемые слои и контракты.

## 2. Определение scope

- Зафиксировать, какие части системы меняются.
- Зафиксировать, какие части системы не должны затрагиваться.
- Определить допустимый объём сопутствующего рефакторинга.

## 3. Проектирование решения

- Выбрать подход.
- Оценить архитектурные последствия.
- Оценить риски регрессий, безопасности и эксплуатации.
- Определить, какие тесты нужно изменить или добавить.

## 4. Реализация

- Вносить изменения строго в рамках scope.
- Не смешивать несколько разных задач в одном изменении.
- Соблюдать архитектурные и кодовые стандарты проекта.

## 5. Тестирование

- Добавить и обновить unit tests.
- Добавить и обновить integration tests.
- Добавить и обновить e2e/UI tests, если это требуется по сценарию.
- Для багов сначала написать падающий тест, затем исправлять код.

## 6. Самопроверка

- Проверить читаемость решения.
- Проверить соответствие RULES.md и специализированным правилам.
- Проверить отсутствие лишних изменений.
- Проверить готовность к review.

## 7. PR/MR

- Создать PR/MR из правильной ветки.
- Убедиться, что commit history понятна.
- Передать изменение в review.

## 8. Review и CI/CD

- Исправить замечания review.
- Дождаться прохождения обязательных quality gates.
- Не обходить процесс ради скорости.

## 9. Merge

- Merge допустим только после выполнения всех обязательных требований.
- После merge изменение должно быть готово к поддержке, эксплуатации и дальнейшему развитию.
