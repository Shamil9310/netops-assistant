# BACKEND_RULES.md

## Назначение документа

Данный документ определяет специальные правила разработки backend-части системы.

Он дополняет общий `RULES.md` и обязателен для всех backend-компонентов:
- API;
- сервисов;
- бизнес-логики;
- работы с базами данных;
- интеграций;
- фоновых задач.

Цель — обеспечить:
- стабильность backend-систем;
- предсказуемость API;
- корректную архитектуру;
- безопасность;
- масштабируемость;
- удобство сопровождения.

---

## 1. Архитектура backend

Backend должен быть построен по слоистой архитектуре:

- Presentation (API / Controllers)
- Application (Use Cases / Services)
- Domain (Business Logic)
- Infrastructure (DB, API, external services)
- Repository (доступ к данным)

### Жёсткие правила

- Контроллер не содержит бизнес-логики.
- Сервис не содержит SQL.
- Domain не знает про БД, HTTP, ORM.
- Infrastructure не содержит бизнес-правил.
- Repository содержит только доступ к данным, без доменной логики.

## 2. API стандарты

### Общие требования

- API должно быть:
  - предсказуемым;
  - консистентным;
  - расширяемым;
  - версионируемым.

### Обязательные правила

- Использовать явную версию API: `/api/v1/...`
- Ответы должны быть единообразными.
- Формат ошибки должен быть стандартизирован.
- Контракты request/response должны быть типизированы и явно описаны.
- Нельзя возвращать случайные структуры ответа для однотипных сценариев.

### HTTP статус-коды

- 200 — успех
- 201 — создано
- 400 — ошибка клиента
- 401 — не авторизован
- 403 — запрещено
- 404 — не найдено
- 409 — конфликт
- 422 — ошибка валидации
- 500 — серверная ошибка

## 3. Схемы и валидация

- Все входные данные валидируются.
- Используются явные схемы.
- Нельзя работать с “сырыми” данными внутри use case без предварительной проверки.

### Обязательно

- валидация типов;
- валидация форматов;
- валидация бизнес-ограничений;
- нормализация данных;
- разделение входных и выходных схем.

## 4. Работа с БД

### Общие правила

- SQL-запросы не размещаются в сервисах.
- Все операции идут через repository слой или специально выделенный data access слой.
- ORM/Query builder должен быть изолирован от бизнес-логики.

### Обязательно

- индексы для часто используемых запросов;
- отсутствие N+1;
- контроль объёма выборок;
- пагинация для списков;
- явная работа с сортировкой и фильтрацией;
- учёт конкурентного доступа, если он важен для сценария.

## 5. Транзакции

- Транзакции должны быть явными.
- Нельзя полагаться на “магическое” поведение ORM.

### Обязательно

- атомарность операций;
- корректный rollback;
- отсутствие частично записанных данных;
- явное определение границ транзакции;
- понимание, какая операция должна быть в одной транзакции, а какая нет.

## 6. Repository слой

Repository:
- изолирует доступ к данным;
- не содержит бизнес-логики;
- возвращает понятные структуры или доменные объекты по правилам проекта.

### Запрещено

- бизнес-правила в repository;
- сложные ветвления, не относящиеся к доступу к данным;
- знание прикладного workflow;
- смешение чтения, записи и бизнес-оркестрации в одном месте без причины.

## 7. Сервисы и use cases

Сервис — это точка реализации бизнес-сценария.

### Сервис обязан

- быть читаемым;
- быть тестируемым;
- иметь одну ответственность;
- координировать зависимости через понятные интерфейсы;
- не зависеть от инфраструктуры напрямую, если это можно изолировать.

### Запрещено

- SQL внутри сервиса;
- HTTP-клиент внутри сервиса без адаптера;
- смешение нескольких use cases;
- хранение временного технического мусора;
- скрытые сайд-эффекты.

## 8. Интеграции

Любая внешняя система должна быть:

- изолирована;
- обёрнута в адаптер;
- не протекать в бизнес-логику;
- заменяема в тестах.

### Обязательно

- таймауты;
- контролируемый retry;
- обработка ошибок;
- логирование;
- защита от нестабильности внешней системы;
- понятные контракты адаптера.

## 9. Очереди и фоновые задачи

### Обязательно

- задачи должны быть идемпотентными;
- должны быть retry-safe;
- должны корректно переживать повторный запуск;
- ошибки должны быть диагностируемыми;
- должна быть понятна политика повторов и фатальных падений.

### Запрещено

- скрытые сайд-эффекты;
- неконтролируемые повторные вызовы;
- отсутствие корреляции с бизнес-событием;
- фоновые задачи без тестов для критичных сценариев.

## 10. Обработка ошибок API

- Ошибки должны быть стандартизированы.
- Пользователь не должен видеть stack trace.
- Формат ошибок должен быть единым для всех API endpoints.

### Обязательно

- понятные error codes;
- разделение:
  - user errors;
  - business errors;
  - system errors;
- безопасные сообщения наружу;
- диагностически полезные сообщения внутрь системы.

## 11. Идемпотентность

Критичные операции должны быть:

- идемпотентными;
- защищёнными от повторного выполнения;
- устойчивыми к сетевым повторам, повторным сообщениям и гонкам, если это применимо.

Особенно это касается:
- создания сущностей;
- платёжных и списывающих сценариев;
- интеграций;
- фоновых задач;
- повторяемых webhook-обработчиков.

## 12. Логирование и observability

### Обязательно

- логировать входящие запросы;
- логировать ошибки;
- логировать внешние вызовы;
- логировать критичные бизнес-события;
- логировать отказоустойчивые сценарии и деградации;
- не логировать секреты и чувствительные данные.

### Желательно как стандарт зрелого проекта

- correlation id;
- tracing;
- метрики;
- health checks;
- audit-события для критичных действий.

## 13. Производительность backend

- избегать лишних запросов;
- контролировать сложность алгоритмов;
- использовать кэширование, если оно оправдано;
- не выполнять тяжёлую работу в request/response цикле без необходимости;
- учитывать рост нагрузки и объёма данных заранее.

## 14. Безопасность backend

### Обязательно

- проверка входных данных;
- защита от injection;
- контроль доступа;
- rate limiting там, где это необходимо;
- audit-логирование критичных действий;
- принцип минимально необходимых прав;
- защита внутренних endpoint и технических операций.

## 15. Миграции

### Обязательно

- миграции должны быть:
  - воспроизводимыми;
  - управляемыми;
  - безопасными для production;
  - проверяемыми на rollback-сценарии там, где это возможно.

### Запрещено

- ручные изменения БД без миграций;
- неописанные schema changes;
- смешение миграции схемы и скрытых data-fix без явного описания.

## 16. Главный принцип backend

Backend должен быть:

- предсказуемым;
- устойчивым к ошибкам;
- безопасным;
- масштабируемым;
- понятным без автора;
- удобным для сопровождения и диагностики.
