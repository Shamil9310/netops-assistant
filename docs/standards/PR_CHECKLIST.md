# PR_CHECKLIST.md

## Назначение документа

Данный документ содержит обязательный чек-лист для разработчика перед открытием Pull Request / Merge Request.

Цель — гарантировать, что изменения:
- соответствуют архитектуре;
- протестированы;
- безопасны;
- готовы к review;
- не создают скрытых проблем в production.

Pull Request не должен открываться, пока не выполнены все пункты данного чек-листа.

---

## 1. По задаче

- Цель задачи понятна и сформулирована.
- Scope изменений определён и не нарушен.
- В PR/MR нет несвязанных изменений.
- Сопутствующий рефакторинг ограничен и оправдан.

## 2. По архитектуре

- Архитектурные слои не смешаны.
- Бизнес-логика не протекла в presentation/UI слой.
- Инфраструктурные детали не протекли в domain/business слой.
- Изменения не ломают существующие контракты без явной причины.

## 3. По коду

- Код соответствует стандартам проекта.
- Новый код типизирован.
- Имена сущностей понятны и самодокументируемы.
- Нет спагетти-кода, магических значений и скрытых зависимостей.
- Нет временных решений без явной фиксации.
- Нет технического мусора, dead code и закомментированных фрагментов “на потом”.

## 4. По комментариям и документации

- Докстринги и комментарии написаны на русском языке.
- Комментарии объясняют причину, ограничения и бизнес-смысл.
- Обновлена документация, если без неё изменение будет непонятно коллегам.
- Если зафиксирована новая идея, она добавлена в `docs/PRODUCT_BACKLOG.md`.

## 5. По ошибкам и логированию

- Ошибки обрабатываются явно.
- Исключения не проглатываются.
- Нет `except Exception` без веского основания.
- Логирование добавлено там, где это требуется.
- В логи не попадают секреты и чувствительные данные.
- `print()` в production/backend-коде не используется.

## 6. По безопасности

- В коде и репозитории нет секретов.
- Конфигурация не содержит небезопасных дефолтов.
- Входные данные валидируются.
- Нет небезопасных shortcut-решений.

## 7. По тестам

- Добавлены или обновлены unit tests.
- Добавлены или обновлены integration tests, если затронуты взаимодействия компонентов.
- Добавлены или обновлены e2e/UI tests, если затронут критичный пользовательский сценарий.
- Покрыты happy path, edge cases, некорректный ввод и ошибки.
- Для исправления бага сначала добавлен падающий тест.
- Достигнуты:
  - 95% line coverage;
  - 95% branch coverage.

## 8. По готовности к review

- PR/MR читается логически и последовательно.
- Изменения атомарны и объяснимы.
- Commit messages соответствуют стандарту `type: description`.
- Ветка названа по стандарту проекта.
- CI/CD готов к запуску без заведомо падающих проверок.
