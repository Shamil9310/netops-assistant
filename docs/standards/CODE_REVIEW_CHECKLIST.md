# CODE_REVIEW_CHECKLIST.md

## Назначение документа

Данный документ определяет обязательный чек-лист для проведения code review.

Цель — обеспечить:
- высокое качество кода;
- корректность архитектуры;
- отсутствие регрессионных рисков;
- соответствие стандартам проекта;
- безопасность и поддерживаемость решений.

Review не является формальностью и должен выполняться строго по данному чек-листу.

---

## 1. Понимание задачи

- Решение действительно закрывает задачу.
- Scope не нарушен.
- Нет скрытого расширения требований.
- Нет лишних изменений вне цели задачи.

## 2. Архитектура

- Слои не смешаны.
- Границы ответственности сохранены.
- Нет избыточной связанности.
- Нет утечек инфраструктурной логики в бизнес-слой.
- Решение масштабируемо и поддерживаемо.

## 3. Качество кода

- Код читаемый и понятный без автора.
- Имена отражают суть.
- Нет избыточной сложности.
- Нет ненужной вложенности.
- Нет дублирования, которое стоило устранить.
- Нет скрытых зависимостей и магических значений.

## 4. Корректность решения

- Учтены happy path и негативные сценарии.
- Учтены граничные случаи.
- Учтены сценарии ошибок.
- Нет очевидных регрессионных рисков.
- Нет нарушения обратной совместимости без причины.

## 5. Ошибки и логирование

- Ошибки обрабатываются явно и корректно.
- Диагностические сообщения понятны.
- Логирование полезно для эксплуатации.
- Секреты и чувствительные данные не логируются.

## 6. Безопасность

- Нет хардкода секретов.
- Нет небезопасных значений по умолчанию.
- Нет очевидных security-антипаттернов.
- Внешние входные данные валидируются и ограничиваются.

## 7. Тесты

- Тесты действительно проверяют поведение, а не создают иллюзию покрытия.
- Есть позитивные сценарии.
- Есть негативные сценарии.
- Есть edge cases.
- Для багфикса есть тест, воспроизводящий дефект.
- Покрытие соответствует требованиям проекта.

## 8. Эксплуатация и сопровождение

- Решение удобно сопровождать.
- Поведение понятно для следующего разработчика.
- Есть необходимые комментарии и докстринги.
- При необходимости обновлена документация.

## 9. Итог review

Review может быть одобрен только если:
- решение корректно;
- архитектура не ухудшена;
- код поддерживаем;
- тесты достаточны;
- CI/CD должен быть способен пройти без обходных путей.
