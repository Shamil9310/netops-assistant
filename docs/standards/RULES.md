# RULES.md

## Назначение документа

Данный документ определяет обязательные правила разработки, архитектуры,
процесса работы над задачами, качества кода, тестирования, review и поставки изменений.

Документ является единым стандартом для всех проектов:
backend, frontend, fullstack, CLI, автоматизации и внутренних сервисов.

Все правила являются обязательными и не допускают исключений.

---

## 1. Общие принципы

- Мы пишем production-grade код, а не pet-проекты.
- Любое изменение должно быть:
  - понятным;
  - поддерживаемым;
  - тестопригодным;
  - безопасным;
  - предсказуемым в эксплуатации;
  - удобным для коллег, ревьюеров и тестировщиков.
- Код пишется с расчётом на долгосрочную поддержку, развитие и передачу другому разработчику без устных пояснений.
- В спорных случаях приоритет имеют:
  1. безопасность;
  2. предсказуемость поведения;
  3. поддерживаемость;
  4. читаемость;
  5. удобство расширения;
  6. скорость разработки.

## 2. Обязательный технологический стандарт

- Обязательно придерживаемся PEP 8.
- Используем Python 3.14 там, где проект включает Python.
- Для всего нового Python-кода обязательна полная типизация (`type hints`).
- Для frontend- и fullstack-проектов код также должен соответствовать строгим стандартам типизации и качества, принятым в используемом стеке.
- Используем только понятные, самодокументируемые имена переменных, функций, классов, модулей и файлов.
- Сокращения, неочевидные аббревиатуры и размытые названия запрещены.
- Магические числа, строки, флаги и скрытые зависимости запрещены.
- Одна функция, один метод, один модуль и один компонент должны иметь одну чёткую ответственность.
- Спагетти-код, чрезмерная вложенность и смешение нескольких уровней абстракции в одном месте запрещены.

## 3. Язык документации и комментариев

- Докстринги и комментарии к коду пишем на русском языке.
- Комментарии должны быть:
  - подробными;
  - понятными;
  - полезными для любого уровня разработчика — от junior до senior.
- Комментарии должны объяснять:
  - бизнес-логику;
  - причину решения;
  - архитектурные ограничения;
  - нетривиальные сценарии;
  - риски и компромиссы.
- Комментарии не должны дублировать очевидный код.
- Если код можно сделать понятнее именами и структурой — сначала улучшаем код, а не компенсируем плохой код лишними комментариями.
- Публичные сущности обязаны иметь корректные и содержательные докстринги.

## 4. Архитектурные правила

Архитектура должна быть слоистой и предсказуемой. Слои не должны смешиваться.

### Обязательное разделение по слоям

- **Presentation / UI layer**  
  Отвечает только за отображение, пользовательский ввод, обработку запросов, формирование ответов и передачу данных дальше.

- **Application / Use Case layer**  
  Содержит сценарии выполнения, orchestration, прикладные правила и координацию работы домена, инфраструктуры и внешних компонентов.

- **Domain / Business Logic layer**  
  Содержит бизнес-правила, доменные модели, инварианты и основную логику предметной области.

- **Infrastructure layer**  
  Содержит интеграции с внешними системами, БД, API, брокерами, файлами, кэшем, очередями, сетевыми сервисами и другими техническими зависимостями.

- **Data Access / Repository layer**  
  Содержит доступ к данным и изоляцию механизма хранения от бизнес-логики.

### Запрещено

- смешивать бизнес-логику с API-слоем;
- писать SQL/ORM-логику внутри бизнес-сервисов;
- тянуть инфраструктурные детали в domain;
- смешивать схемы, transport objects и внутренние доменные модели без явной причины;
- завязывать ключевую логику напрямую на framework-specific код, если это можно изолировать.

## 5. Обязательный порядок работы над задачей

Работа над задачей выполняется только в следующем порядке:

1. Анализ задачи
2. Уточнение ограничений и контекста
3. Определение scope изменений
4. Выбор решения
5. Оценка рисков и побочных эффектов
6. Реализация
7. Написание и обновление тестов
8. Самопроверка
9. Подготовка изменений к review
10. Прохождение review
11. Прохождение CI/CD
12. Merge по правилам процесса

Писать код до анализа задачи, границ изменения и выбора решения запрещено.

## 6. Scope изменений

Перед реализацией обязательно определяется scope задачи:

- какие файлы и слои затрагиваются;
- какие контракты затрагиваются;
- какие части системы затрагивать нельзя;
- где заканчивается текущая задача;
- какие риски есть у побочных изменений.

### Правила работы со scope

- Запрещено бесконтрольно расширять задачу.
- Разрешён только ограниченный сопутствующий рефакторинг, если он:
  - напрямую связан с текущей задачей;
  - уменьшает риск;
  - упрощает поддержку;
  - не раздувает scope до неуправляемого состояния;
  - не ломает лишние контракты.
- Масштабный рефакторинг должен выноситься в отдельную задачу.
- Изменения должны быть локальными, осознанными и проверяемыми.

## 7. Правила изменения существующего кода

- Нельзя ломать существующие контракты без явной задачи и согласованного решения.
- Обратная совместимость должна сохраняться везде, где это возможно и не противоречит целям задачи.
- Рефакторинг не должен скрывать функциональные изменения.
- Если в задаче есть и функциональные изменения, и рефакторинг, они должны быть логически разделены.
- Любое изменение должно быть минимально достаточным для решения задачи, но не минималистичным в ущерб качеству.

## 8. Git и модель ветвления

Используется жёсткая модель ветвления уровня бизнес-проектов.

### Обязательные правила

- Новая функциональность всегда разрабатывается в отдельной ветке.
- Интеграция идёт через `develop`.
- Стабильная ветка — `main`.
- Прямые коммиты в `develop` запрещены.
- Прямые коммиты в `main` запрещены.
- Любое продвижение изменений между этапами должно идти через формализованный CI/CD-процесс.

### Именование веток

Допустимы только следующие префиксы:

- `feature/...` — новая функциональность
- `fix/...` — исправления
- `refactor/...` — рефакторинг
- `docs/...` — документация
- `test/...` — изменения в тестах

### Требования к имени ветки

Имя ветки после префикса должно быть:

- конкретным;
- понятным;
- отражающим суть задачи.

Примеры корректных имён:

- `feature/add-device-import`
- `fix/journal-date-filter`
- `refactor/split-ipam-service`

Примеры запрещённых имён:

- `feature/test`
- `fix/bug`
- `feature/newstuff`

## 9. Коммиты

### Общие правила

- Коммиты должны быть атомарными.
- Один коммит должен отражать одну логически цельную часть изменений.
- Нельзя смешивать в одном коммите несвязанные изменения.
- Сообщения коммитов должны быть понятными по смыслу.

### Обязательный формат commit message

Используется только формат:

`type: description`

### Допустимые типы коммитов

- `feat`
- `fix`
- `refactor`
- `test`
- `docs`
- `chore`
- `style`
- `perf`
- `build`
- `ci`
- `revert`

### Примеры

- `feat: add device import validation`
- `fix: handle empty journal response`
- `refactor: split ipam allocation service`
- `test: add negative cases for auth service`
- `docs: update deployment notes`

### Запрещено

- `fix`
- `update`
- `test`
- `123`
- `misc`
- любые бессодержательные сообщения

## 10. Pull Request / Merge Request

- Попадание изменений в `develop` допускается только через Pull Request / Merge Request.
- Прямые push и прямые merge в `develop` запрещены.
- Merge допускается только после прохождения всех обязательных этапов качества и review.
- Автор может выполнять final merge только после выполнения всех обязательных условий процесса.

## 11. Code Review

Code review обязателен для любого merge.

### Review должен быть строгим и полноценным

Проверяются:

- архитектура;
- читаемость;
- соблюдение RULES.md;
- корректность решения;
- тесты;
- риски побочных эффектов;
- поддерживаемость;
- безопасность;
- эксплуатационные последствия;
- качество границ между слоями;
- корректность обработки ошибок;
- корректность логирования;
- отсутствие технического мусора и временных решений.

### Approval

Для merge обязательны 2 approval:

- от senior developer;
- от lead developer.

Без двух таких подтверждений merge запрещён.

### Замечания review

- Все замечания review должны быть устранены до merge.
- Игнорирование замечаний review запрещено.
- Формальный review “для галочки” запрещён.

## 12. CI/CD и quality gates

Merge невозможен без прохождения обязательных quality gates в CI/CD.

### Обязательные проверки

- линтинг;
- форматирование;
- типизация;
- тесты;
- coverage;
- проверки докстрингов;
- security checks;
- dependency / build / pipeline checks, если применимо к проекту.

### Merge запрещён, если

- хотя бы один обязательный pipeline job упал;
- есть красные quality gates;
- есть неустранённые замечания review;
- не выполнены требования RULES.md;
- нет обязательного покрытия тестами.

## 13. Тестирование

Тестирование обязательно для любого нового поведения, исправления багов, изменения логики и критичных рефакторингов.

### Обязательное покрытие

Минимальный порог покрытия:

- 95% line coverage
- 95% branch coverage

Coverage ниже порога означает, что задача не готова.

### Обязательные уровни тестов

Тесты должны быть разделены по уровням:

- unit tests — проверяют изолированную логику;
- integration tests — проверяют взаимодействие слоёв, БД, API, очередей, файловой системы и интеграций;
- end-to-end / UI tests — проверяют критичные пользовательские сценарии.

### Что обязаны покрывать тесты

- happy path;
- edge cases;
- некорректный ввод;
- сценарии ошибок;
- граничные состояния;
- регрессионные сценарии;
- критичные ветки логики.

### Правило исправления багов

Любой баг сначала воспроизводится тестом, и только потом исправляется код.

Порядок обязателен:

1. создаётся тест, который падает и подтверждает дефект;
2. вносится исправление;
3. тест становится зелёным;
4. добавляются недостающие сопутствующие проверки, если нужно закрыть регрессионный риск.

## 14. Обработка ошибок

Обработка ошибок должна быть явной, предсказуемой и диагностируемой.

### Обязательные правила

- исключения нельзя молча проглатывать;
- `except Exception` без обоснованной причины запрещён;
- ошибка должна либо:
  - корректно обрабатываться,
  - либо явно пробрасываться выше;
- бизнес-ошибки, пользовательские ошибки и системные ошибки должны быть разделены;
- тексты ошибок должны быть понятными и полезными для диагностики;
- ошибки не должны терять контекст;
- fallback-поведение должно быть явным, а не скрытым.

### Запрещено

- silent fail;
- пустые `except`;
- скрытые retry без контроля;
- возврат некорректного “успешного” результата при ошибке;
- проглатывание исключения ради “чтобы не падало”.

## 15. Логирование

Логирование обязательно и должно быть пригодным для эксплуатации.

### Обязательные правила

- `print()` в backend- и production-коде запрещён;
- используется нормальный механизм логирования;
- уровни логов обязательны:
  - `DEBUG`
  - `INFO`
  - `WARNING`
  - `ERROR`
  - `CRITICAL`
- ключевые операции должны логироваться с достаточным контекстом;
- ошибки логируются с диагностически полезной информацией;
- логирование не должно шуметь и не должно превращаться в мусорный поток.

### Строго запрещено логировать

- пароли;
- токены;
- ключи;
- секреты;
- чувствительные персональные данные;
- приватные данные, которые не нужны для диагностики.

## 16. Секреты и конфигурация

Работа с конфигурацией и секретами строго регламентирована.

### Обязательные правила

- секреты никогда не хранятся в коде и репозитории;
- используется:
  - environment variables,
  - secret storage,
  - защищённые механизмы поставки конфигурации;
- небезопасные значения по умолчанию запрещены;
- конфигурации `dev`, `stage`, `prod` должны быть разделены явно;
- конфигурация должна быть прозрачной и предсказуемой;
- значения по умолчанию не должны создавать скрытых рисков безопасности.

## 17. Работа с зависимостями

- Зависимости добавляются только при реальной необходимости.
- Каждая новая зависимость должна быть оправдана.
- Нельзя тянуть библиотеку ради одной тривиальной функции, если это разумно решается стандартными средствами.
- Версии зависимостей должны быть управляемыми и предсказуемыми.
- Устаревшие, небезопасные и неподдерживаемые зависимости использовать запрещено.
- Изменения зависимостей должны проходить через review и CI/CD.

## 18. Безопасность

Безопасность не является опциональной.

### Обязательные принципы

- не хранить секреты в коде;
- не логировать чувствительные данные;
- валидировать входные данные;
- не доверять внешним источникам без проверки;
- минимизировать привилегии;
- избегать небезопасных дефолтов;
- явно контролировать доступ, роли и ограничения;
- исключать несанкционированные побочные эффекты.

## 19. Производительность и масштабируемость

- Производительность должна учитываться на этапе проектирования, а не только после проблем в production.
- Критичные по нагрузке участки должны проектироваться осознанно.
- Не допускаются заведомо неэффективные решения без явной причины.
- Оптимизация должна быть обоснованной, измеримой и не разрушать читаемость без необходимости.
- Для оптимизаций используется тип коммита `perf`.

## 20. Документация изменений

Любое важное изменение должно быть сопровождено обновлением документации, если без этого другой разработчик, тестировщик или инженер сопровождения не сможет корректно понять:

- что изменилось;
- зачем это сделано;
- как это использовать;
- какие ограничения появились;
- какие риски или особенности нужно знать.

Если пользователь начинает сообщение с фразы **«у меня идея»**, эта идея обязательно фиксируется в:

`docs/PRODUCT_BACKLOG.md`

## 21. Definition of Done (DoD)

Задача считается завершённой только если одновременно выполнены все условия ниже:

- код соответствует всем правилам этого документа;
- выполнен обязательный порядок работы над задачей;
- scope изменений определён и соблюдён;
- архитектурные слои не нарушены;
- изменения читаемы, поддерживаемы и понятны;
- добавлены или обновлены тесты;
- покрыты позитивные и негативные сценарии;
- покрыты edge cases и ошибки;
- соблюдён минимальный порог покрытия:
  - 95% line coverage;
  - 95% branch coverage;
- баги сначала воспроизводятся тестами, затем исправляются;
- нет хардкода секретов;
- нет небезопасных дефолтов;
- ошибки обрабатываются явно;
- логирование реализовано корректно;
- комментарии и докстринги отражают причины, бизнес-смысл и ограничения;
- CI/CD полностью зелёный;
- quality gates пройдены;
- review пройден;
- получены 2 обязательных approval:
  - senior developer;
  - lead developer;
- изменения готовы к поддержке другим разработчиком без устных пояснений.

## 22. Что запрещено

Запрещено:

- писать код без анализа задачи;
- писать код вне определённого scope;
- смешивать слои архитектуры;
- писать нетипизированный новый код;
- писать непонятный или неструктурированный код;
- использовать бессодержательные имена;
- оставлять магические значения;
- делать silent fail;
- проглатывать исключения;
- использовать `print()` в production/backend-коде;
- хранить секреты в коде;
- обходить тесты;
- снижать требования к quality gates;
- делать merge без review;
- делать merge без CI/CD;
- обходить обязательный процесс ради скорости;
- оставлять временные решения без явной фиксации и отдельного плана устранения;
- считать задачу готовой, если выполнена только “основная логика”, но не закрыты эксплуатационные, тестовые и архитектурные требования.

## 23. Главный принцип

Мы пишем код так, чтобы он был:

- безопасен в production;
- понятен без автора;
- стабилен под нагрузкой и изменениями;
- удобен для review;
- удобен для тестирования;
- удобен для сопровождения;
- удобен для развития в долгую.

Скорость без качества не считается результатом.  
Рабочим считается только тот результат, который проходит инженерные, архитектурные, эксплуатационные и процессные требования.
