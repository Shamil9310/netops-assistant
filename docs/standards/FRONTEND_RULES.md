# FRONTEND_RULES.md

## Назначение документа

Данный документ определяет специальные правила разработки frontend-части системы.

Он дополняет общий `RULES.md` и обязателен для всех пользовательских интерфейсов:
- web-приложений;
- SPA;
- админ-панелей;
- внутренних интерфейсов;
- интеграционных UI.

Цель — обеспечить:
- предсказуемый и стабильный UI;
- единый UX;
- корректную работу с backend;
- удобство поддержки и масштабирования;
- высокое качество пользовательского опыта.

---

## 1. Архитектура frontend

Frontend должен быть структурированным и предсказуемым.

### Обязательные принципы

- разделение на слои:
  - UI (presentation)
  - state management
  - API слой
  - прикладная логика
- логика не должна “размазываться” по компонентам;
- бизнес-правила не должны быть захардкожены в случайных UI-узлах.

### Запрещено

- писать бизнес-логику прямо в UI-компонентах;
- делать “толстые” компоненты с множеством ответственностей;
- смешивать API-вызовы и рендеринг в одном месте без причины;
- строить интерфейс вокруг неявных побочных эффектов.

## 2. Структура компонентов

Компоненты должны быть:

- маленькими;
- переиспользуемыми;
- предсказуемыми;
- легко тестируемыми.

### Обязательные правила

- один компонент — одна ответственность;
- большие компоненты должны разбиваться на подкомпоненты;
- логика выносится в hooks / services / store, если она разрастается;
- UI-компоненты должны быть максимально “глупыми”, если это не container-компонент.

## 3. Container / Presentational подход

Разделение обязательно:

- **Container (smart)**:
  - получает данные;
  - управляет состоянием;
  - вызывает API;
  - содержит orchestration-логику;

- **Presentational (dumb)**:
  - отображает данные;
  - получает зависимости через props;
  - не знает про API;
  - не содержит бизнес-оркестрации.

Смешение этих ролей допустимо только для действительно маленьких и локальных сценариев.

## 4. Управление состоянием

### Обязательные правила

- состояние должно быть:
  - централизованным там, где это действительно необходимо;
  - локальным там, где оно не нужно глобально;
  - предсказуемым;
  - минимальным.
- нельзя дублировать состояние;
- нельзя хранить derived state без необходимости;
- изменения состояния должны быть понятны и отслеживаемы.

### Запрещено

- хаотичное использование state;
- смешивание локального и глобального состояния без причины;
- side effects внутри render;
- скрытые зависимости между компонентами через неявное состояние.

## 5. Работа с API

### Обязательные правила

- все API-вызовы должны быть вынесены в отдельный слой;
- UI не должен напрямую работать с `fetch`, `axios` и аналогами;
- API-клиент должен быть типизирован;
- ответы должны нормализоваться, если это требуется проектом;
- обработка ошибок API должна быть централизованной или хотя бы единообразной.

### Обязательная обработка состояний

Каждый сетевой сценарий обязан учитывать:

- loading;
- error;
- empty state;
- success.

## 6. Loading / Error / Empty states

Каждый запрос обязан корректно обрабатывать:

- loading state;
- error state;
- empty state.

### Запрещено

- “белый экран”;
- отсутствие индикации загрузки;
- молчаливое игнорирование ошибок;
- неочевидное поведение при пустых данных.

## 7. Формы и валидация

### Обязательные правила

- валидация на frontend обязательна;
- ошибки должны быть понятны пользователю;
- форма не должна отправляться с очевидно некорректными данными;
- структура ошибок должна быть единообразной.

### Обязательно

- синхронная валидация;
- асинхронная валидация, если этого требует сценарий;
- отображение ошибок рядом с полями;
- защита от повторной отправки, если сценарий этого требует;
- корректная работа disabled / loading состояний кнопок и полей.

## 8. UX и консистентность

UI должен быть:

- единообразным;
- предсказуемым;
- понятным;
- логичным по сценариям поведения.

### Обязательно

- одинаковое поведение одинаковых элементов;
- единые паттерны взаимодействия;
- отсутствие “сюрпризов” для пользователя;
- единый тон текста интерфейса;
- понятные подписи, кнопки, статусы и сообщения.

## 9. Accessibility (A11y)

### Обязательно

- использование семантических HTML-элементов;
- поддержка клавиатурной навигации;
- корректные aria-атрибуты;
- читаемость для screen reader;
- достаточная различимость состояний;
- отсутствие критически важных действий, доступных только мышью.

## 10. Dark theme

Тёмная тема обязательна.

### Правила

- все новые экраны и компоненты обязаны корректно работать в тёмной теме;
- нельзя добавлять компоненты, которые визуально ломают тему;
- состояния hover, focus, disabled, error и success должны быть читаемыми в тёмной теме;
- контраст текста и элементов должен быть достаточным.

## 11. Оптимизация рендера

### Обязательно

- избегать лишних перерендеров;
- использовать мемоизацию там, где это оправдано;
- разделять тяжёлые компоненты;
- не рендерить большие объёмы данных без стратегии оптимизации.

### Запрещено

- premature optimization без причины;
- игнорирование очевидных проблем производительности;
- создание сложных вычислений внутри render без необходимости.

## 12. Frontend тесты

### Обязательные уровни

- unit tests — логика, утилиты, hooks;
- component tests — поведение компонентов;
- integration tests — взаимодействие компонентов, состояния и API-слоя;
- e2e tests — критичные пользовательские сценарии.

### Обязательно

- тестировать поведение, а не внутреннюю реализацию;
- покрывать:
  - happy path;
  - ошибки;
  - edge cases;
  - пустые состояния;
  - сценарии доступа и ограничений, если они есть.

## 13. E2E тесты

Обязательны для:

- авторизации;
- критичных пользовательских сценариев;
- ключевых бизнес-процессов;
- сценариев, где ошибка особенно дорога для пользователя или бизнеса.

E2E не заменяют unit и integration tests, а дополняют их.

## 14. Безопасность frontend

### Обязательно

- защита от XSS;
- не хранить токены в небезопасных местах без обоснования;
- не доверять данным с backend без проверки контракта;
- не логировать чувствительные данные;
- не оставлять отладочные механизмы в production-сборке.

## 15. Работа с токенами и auth

### Обязательно

- централизованное управление auth;
- обработка:
  - истечения токена;
  - logout;
  - refresh token;
- защита приватных роутов;
- корректное поведение UI при потере сессии;
- отсутствие случайного “частично авторизованного” состояния.

## 16. Обработка ошибок UI

Ошибки должны:

- быть понятны пользователю;
- не ломать интерфейс;
- не приводить к зависанию приложения;
- не создавать ложного ощущения успешного действия.

Для критичных зон должны использоваться error boundaries или аналогичные защитные механизмы, если стек это поддерживает.

## 17. Производительность frontend

### Обязательно

- lazy loading там, где это оправдано;
- code splitting;
- минимизация bundle;
- контроль загрузки ресурсов;
- отсутствие тяжёлых библиотек без веской причины;
- понимание стоимости рендера и сетевых запросов.

## 18. Главный принцип frontend

Frontend должен быть:

- стабильным;
- предсказуемым;
- удобным;
- быстрым;
- понятным без автора;
- единообразным;
- готовым к долгосрочному развитию и сопровождению.
