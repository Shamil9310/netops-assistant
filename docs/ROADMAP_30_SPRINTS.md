# NetOps Assistant — roadmap на 30 спринтов

Ниже — целевой план доведения проекта до рабочего внутреннего production-уровня.

## Статусы

- ✅ — завершён
- 🔄 — в работе
- ⬜ — не начат

---

## ✅ Sprint 01 — Product reset и фиксация scope

- обновить README и product docs;
- зафиксировать роли `employee / manager / developer`;
- зафиксировать контуры:
  - `day ops`
  - `night works`
  - `reports`
  - `templates`
  - `developer dashboard`;
- определить MVP и non-MVP;
- зафиксировать, что основа проекта — **журнал записей**, а не только отчёты.

**Результат спринта:**
- зафиксирована продуктовая рамка проекта;
- определено, что система покрывает и дневную работу, и ночные работы, и отчётность.

---

## ✅ Sprint 02 — Config and environments baseline

- разделить `dev / test / prod` конфиги;
- убрать небезопасные дефолты из non-dev;
- пересобрать env-модель frontend и backend;
- исправить design подключения backend/frontend;
- привести запуск локального контура к воспроизводимому виду.

**Результат спринта:**
- проект можно стабильно запускать локально;
- окружения разделены концептуально.

---

## ✅ Sprint 03 — Database foundation

- подключить Alembic;
- создать базовую миграционную цепочку;
- убрать `create_all()`;
- описать `seed / bootstrap` для dev-среды;
- стабилизировать локальную PostgreSQL-схему.

**Результат спринта:**
- БД живёт через миграции;
- приложение больше не зависит от автогенерации схемы на старте.

---

## ✅ Sprint 04 — Auth core v1

- привести локальную auth в рабочее состояние;
- настроить session lifecycle;
- реализовать logout/revoke;
- добавить audit events для auth;
- починить стабильный вход в локальном режиме.

**Результат спринта:**
- пользователь может войти в систему;
- локальный auth-контур работает end-to-end.

---

## ✅ Sprint 05 — Roles and RBAC foundation

- зафиксировать роли:
  - `employee`
  - `manager`
  - `developer`;
- определить permission policies;
- описать team visibility rules;
- развести backend guards и frontend navigation guards;
- заложить read-only модель руководителя.

**Результат спринта:**
- зафиксирована ролевая модель доступа;
- понятно, кто что видит и кто что может менять.

---

## ✅ Sprint 06 — Teams and users baseline

- определить `Team model`;
- определить `user-team relation`;
- зафиксировать manager scope;
- подготовить базовые `user/team APIs`;
- заложить командный контур для будущего расширения.

**Результат спринта:**
- готов фундамент под многопользовательский режим;
- руководительский и командный сценарии описаны.

---

## ✅ Sprint 07 — Journal domain model v1

- реализовать `ActivityEntry` как главную сущность журнала;
- добавить поля:
  - `user_id`
  - `work_date`
  - `started_at`
  - `ended_at`
  - `activity_type`
  - `title`
  - `description`
  - `ticket_number`
  - `created_at`
  - `updated_at`;
- определить `activity types`;
- определить базовую status-модель записи;
- подготовить repository/service/API слой;
- зафиксировать правило:
  - отчёт строится по `work_date`,
  - а не по дате создания записи.

**Результат спринта:**
- появляется реальная доменная запись журнала;
- можно строить всё остальное вокруг неё.

---

## ✅ Sprint 08 — Journal UI v1 и рабочая дата записи

- оживить кнопку `+ Запись`;
- сделать форму создания записи;
- сделать `work_date` обязательным полем;
- по умолчанию подставлять сегодняшнюю дату;
- разрешить указывать:
  - прошлую дату
  - текущую дату
  - будущую дату;
- сохранить запись в БД;
- отобразить запись в ленте выбранного дня;
- добавить быстрый сценарий:
  - сегодня создать запись за вчера;
  - сегодня создать запись за любой предыдущий день.

**Результат спринта:**
- можно вручную добавлять реальные записи;
- можно вести журнал задним числом и на будущие даты.

---

## ✅ Sprint 09 — Today / Day view on real data

- перевести экран `Сегодня` на реальные данные;
- выводить записи по выбранной `work_date`;
- показывать агрегаты за день;
- строить timeline;
- считать counters по типам и статусам;
- отображать planned events дня;
- убрать хардкодный demo-контент со страницы дня.

**Результат спринта:**
- страница дня становится реальным рабочим экраном, а не макетом.

---

## ✅ Sprint 10 — Edit / delete / validation для записей

- добавить редактирование записи;
- добавить удаление записи;
- сделать валидацию:
  - дата обязательна
  - заголовок обязателен
  - время начала не позже времени окончания;
- добавить понятные ошибки формы;
- покрыть CRUD pytest-тестами;
- пометить записи, добавленные задним числом.

**Результат спринта:**
- журнал становится пригоден для повседневной ручной работы.

---

## ✅ Sprint 11 — Planned events

- реализовать `PlannedEvent model`;
- сделать ручное создание будущих звонков и событий;
- позволить вносить заранее подготовленные TrueConf/созвоны;
- добавить auto-include logic в дашборд дня;
- сделать CRUD API с фильтрами;
- добавить конвертацию planned event → ActivityEntry.

**Результат спринта:**
- заранее известные активности можно планировать и учитывать в отчётах.

---

## ✅ Sprint 12 — Reports engine v1

- реализовать daily report generator;
- реализовать weekly report generator;
- реализовать range report generator;
- строить unified markdown output;
- группировать записи по типам активности;
- рассчитывать суммарное время;
- использовать в генерации именно `work_date`.

**Результат спринта:**
- появляются реальные отчёты за день, неделю и период.

---

## ✅ Sprint 13 — Reports UI and draft flow

- добавить предпросмотр отчёта;
- добавить выбор периода;
- сделать кнопку `Собрать отчёт`;
- сделать кнопку `Обновить отчёт`;
- ввести режим `draft`;
- показывать, что в отчёт попали записи по выбранной рабочей дате;
- добавить историю генераций.

**Результат спринта:**
- пользователь может собирать и пересобирать черновики отчётов из интерфейса.

---

## ✅ Sprint 14 — Report finalization и изменение задним числом

- ввести сущность `ReportRecord`;
- поддержать статусы:
  - `draft`
  - `final`;
- реализовать `snapshot` текста зафиксированного отчёта;
- после добавления новой записи в уже завершённый день показывать:
  - “после фиксации появились новые записи”;
- разрешить:
  - пересобрать отчёт,
  - либо оставить старую финальную версию;
- зафиксировать поведение для “вчерашний отчёт, новая запись сегодня”.

**Результат спринта:**
- появляется контролируемый контур фиксации отчётов;
- можно безопасно дописывать прошлые дни.

## ✅ Sprint 18 — Execution mode for night works

**Результат спринта:**

# Что должно быть готово к концу Sprint 30

- домен дневного журнала;
- домен ночных работ;
- шаблоны;
- отчёты за день / неделю / период;
- возможность добавлять записи за прошлую, текущую и будущую дату;
- возможность дописывать прошлый день после его первичного формирования;

- developer dashboard;
- LDAP production auth;
- экспорт документов;
- архив и поиск;
- операционная готовность.
