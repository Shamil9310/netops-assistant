# NetOps Assistant — правила кода

Документ обязателен для всех изменений в `backend` и `frontend`.

## 1. Базовые принципы

- Пишем код для поддержки, а не только для прохождения текущей задачи.
- Любой endpoint, сервис или UI-экран должен быть предсказуемым и расширяемым.
- Минимизируем hidden logic: сложные правила выносим в `services`, а не в роуты и не в UI.
- Изменения должны быть атомарными: один PR = одна понятная цель.

## 2. Структура backend (FastAPI)

- Роуты (`app/api/routes/*`) содержат только:
  - валидацию входа;
  - вызов сервисов;
  - маппинг ответа и HTTP-статусов.
- Бизнес-логика находится в `app/services/*`.
- Работа с БД концентрируется в отдельных слоях доступа (в текущем коде это `services`, в целевом состоянии — `repositories`).
- ORM-модели (`app/models/*`) не должны содержать бизнес-правила уровня сценариев.
- Все новые сущности проходят через явные Pydantic-схемы request/response.

## 3. Структура frontend (Next.js App Router)

- Серверные компоненты используем по умолчанию, клиентские (`"use client"`) — только для интерактива.
- API-вызовы и типы храним в `frontend/lib/*`, не размазываем `fetch` по страницам.
- Страницы (`app/*/page.tsx`) отвечают за композицию экрана, а не за сложные вычисления.
- UI-компоненты в `components/*` должны быть переиспользуемыми и без бизнес-логики backend-уровня.
- Новые маршруты с ограничением доступа обязаны проверять сессию/роль.

## 4. Нейминг и стиль

- Имена переменных, функций, классов и файлов — осмысленные, без сокращений "на удачу".
- Не используем однобуквенные имена, кроме локальных коротких итераторов.
- В Python придерживаемся type hints для публичных функций.
- В TypeScript не отключаем строгую типизацию локальными `any` без причины.
- Комментарии пишем только там, где без них тяжело понять неочевидное решение.

## 5. Работа с данными и БД

- Изменения схемы БД делаем только через миграции (цель: Alembic).
- Нельзя полагаться на `create_all()` как на production-механику миграции.
- Для каждой новой сущности определяем:
  - индексы;
  - ограничения целостности;
  - правила удаления/архивации.
- Даты и время храним в UTC.

## 6. Безопасность (обязательно)

- Секреты не хардкодим и не коммитим.
- Cookie в non-dev окружениях должны быть `secure`.
- Все защищённые endpoint-ы требуют аутентификации и проверок роли (RBAC).
- Для мутаций через cookie-based auth учитываем CSRF-защиту.
- Ошибки авторизации/доступа не должны раскрывать лишние детали.

## 7. API-контракты

- Контракты API должны быть стабильными и явными.
- Любое изменение response-формата фиксируем в PR-описании.
- Используем корректные HTTP-коды (`200/201/204/400/401/403/404/409` и т.д.).
- Ошибки возвращаем в единообразном формате.

## 8. Тестирование и проверка

- Для каждой новой бизнес-функции добавляем или обновляем тесты.
- Минимум перед merge:
  - backend: тесты + линтер (после их внедрения в проект);
  - frontend: сборка + типизация.
- Если тесты временно отсутствуют, в PR явно указываем риски и ручной сценарий проверки.

## 9. Логи и наблюдаемость

- Логи должны помогать расследовать проблему, а не шуметь.
- Не логируем пароли, токены, секреты и чувствительные персональные данные.
- Ошибки в критичных сценариях (auth, export, отчёты) должны иметь диагностический контекст.

## 10. PR-чеклист (короткий)

- Изменение решает конкретную задачу и не тянет лишний рефакторинг.
- Код соответствует слоям и структуре проекта.
- Нет хардкодов секретов и небезопасных дефолтов.
- Обновлена документация, если контракт или поведение изменились.
- Описано, как проверить изменение локально.

## 11. Что запрещено

- Коммитить временные обходы "потом починим" без явного TODO и срока.
- Дублировать бизнес-логику одновременно в backend и frontend.
- Подмешивать крупный рефакторинг в feature-PR без явного запроса.
- Менять публичный API молча, без фиксации в документации и PR.
