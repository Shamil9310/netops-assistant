# Обзор и оценка проекта NetOps Assistant

> Дата анализа: апрель 2026  
> Ветка: develop

---

## 1. Общая оценка

NetOps Assistant — полноценный корпоративный инструмент для инженеров сетевых операций. Стек: FastAPI + PostgreSQL на бэкенде, Next.js 16 + React 19 + TypeScript на фронтенде, CI/CD через GitLab.

**Готовность MVP: ~70%**

- Ядро (журнал, отчёты, ночные работы) — работает
- Вторичные модули (учёба, шаблоны, команда) — частично готовы
- Сквозные вещи (RBAC, тесты, наблюдаемость, обработка ошибок) — отсутствуют или минимальны

---

## 2. Что сделано хорошо

### Бэкенд
- Чёткое разделение слоёв: API → Schemas → Services → Models → DB
- Async/await для всех I/O операций (asyncpg + SQLAlchemy async)
- 24 миграции Alembic с guard-проверкой синхронности схемы
- Комплексная аутентификация: локальная (dev) + LDAP (prod), хеширование argon2, CSRF-защита
- Хороший объём сервисной логики: journal (939 строк), study (766), reports (571), night_work (587)
- Экспорт PDF/DOCX/TXT без внешних зависимостей — грамотное решение
- Принудительный контроль качества в CI: ruff, black, mypy, interrogate, eradicate

### Фронтенд
- Тёмная enterprise-тема, подходящая для ops-команд
- Type-safe API-клиент с TypeScript-типами на все сущности
- 27 компонентов с разумным разделением ответственности

### Инфраструктура
- Docker Compose для локального окружения (3 сервиса: postgres, backend, frontend)
- GitLab CI с 9 стадиями
- Doctor-скрипт для валидации среды разработчика

## 3. Критические баги

### P0 — Потеря данных при рестарте бэкенда
**Файл:** `backend/app/services/auth.py:220-258`

`ensure_bootstrap_user()` при каждом запуске удаляет всех пользователей кроме bootstrap. Один рестарт — потеря всех локальных аккаунтов со всеми данными.

**Исправление:** сделать функцию идемпотентной — создавать bootstrap-пользователя только если он не существует, никого не удалять.

---

### P1 — Пароль в URL
**Файлы:** `frontend/app/api/developer/local-users/route.ts:74-87`, `frontend/app/developer/users/page.tsx:35-39`

Сгенерированный пароль передаётся через query-параметр URL — попадает в историю браузера, логи nginx, referer-заголовки.

**Исправление:** передавать через server-side flash state или session storage.

---

### P1 — PATCH не умеет обнулять поля
**Файлы:** `backend/app/schemas/journal.py:62-82`, `backend/app/services/journal.py:167-190`

Поле `null` в PATCH-запросе игнорируется — нельзя отличить "не передано" от "очистить". Пользователь не может удалить описание, контакт, резолюцию.

**Исправление:** использовать `model_fields_set` Pydantic или явный паттерн с `UNSET`-sentinel.

---

### P1 — Возможны отрицательные интервалы времени
**Файл:** `backend/app/services/journal.py:102-115`

При авто-подстановке `started_at` не проверяется что `started_at < finished_at`. Можно создать запись с инвертированным временем.

---

### P1 — Смена work_date не синхронизирует started_at/finished_at
**Файл:** `backend/app/services/journal.py:167-168, 192-243`

При изменении только `work_date` временны́е поля остаются на старой дате — получаем `work_date=2026-04-11, started_at=2026-04-10`.

## 4. Что отсутствует

### Безопасность
- **RBAC** — роли объявлены в модели (`UserRole`), но middleware-проверки почти нигде не применяются. Большинство эндпоинтов доступны любому авторизованному пользователю
- Нет rate limiting и защиты от брутфорса
- Нет аудит-лога изменений данных

### Тесты
- **Фронтенд: 0 тестов**
- E2E тесты добавлены для критичных пользовательских сценариев
- Интеграционные тесты минимальны
- Покрытие бэкенда: порог 50% — слишком низко для продакшена

### Обработка ошибок
- Нет иерархии кастомных исключений
- Ошибки бизнес-логики бросаются как `ValueError` или `HTTPException` без структуры
- Фронтенд не имеет error boundary

### Наблюдаемость
- Нет структурированного логирования (JSON + correlation ID)
- Нет метрик (Prometheus/Grafana)
- Нет трейсинга
- Нет мониторинга алертов

### Инфраструктура
- Нет NGINX reverse proxy
- Нет SSL/TLS конфигурации
- Нет стратегии бэкапа БД
- Нет health check эндпоинтов для container orchestration

### Фронтенд
- Нет глобального state management — каждая страница фетчит данные независимо
- Нет клиентского кеширования (SWR, React Query)
- Нет валидации форм (Zod/Yup)
- Нет подтверждения деструктивных операций
- Нет автосохранения черновиков

---

## 5. Что можно улучшить

### Архитектура бэкенда
- Сервисы — это наборы функций, не классы. Нет dependency injection. При росте команды это станет проблемой
- SQL-запросы разбросаны по сервисам — нет repository/DAO слоя
- Отсутствует стратегия API versioning beyond `/api/v1`

### UX
- Нет клавиатурных сокращений для ускорения работы
- Нет счётчиков/бейджей в сайдбаре (сколько активных планов, незакрытых задач)
- Нет индикатора несохранённых изменений
- Мобильная адаптация не проверена
- Нет подтверждения при удалении записей

### DX
- Нет генерации OpenAPI-документации (Swagger UI)
- Нет JSDoc-комментариев для frontend-компонентов
- Нет response caching headers на бэкенде
